Colophon

This site has no framework, no bundler, and no build tool that hides a decision from me. It is hand-written HTML and CSS, published on GitHub Pages. That does not mean there is no build system. It means the build system is small, legible, and mine. What follows is how the site is actually made, because the way a thing is engineered is itself a portfolio piece.

The constraint

The homepage and subpages are pure HTML and CSS. No framework, no preprocessor, no bundler, no Google Fonts. The body face is ETBook, the MIT-licensed revival of the type Tufte used in print, self-hosted in two weights. JavaScript appears in exactly four narrow places: a privacy-respecting page-view counter, and three URL-scoped interactive subpages where static markup genuinely cannot express the idea (a Star Ratings cliff simulator, a 4,680-week life grid, and a stochastic epidemic model that runs Python in the browser). Everywhere else, including this page, the rule is HTML and CSS or nothing.

Build pipelines

Five Python scripts turn source files into the published site. Each is idempotent: running it twice with the same inputs produces the same output. Generated files are committed by CI on the same run that triggers them, so the repository is always a faithful snapshot of what is live.

build_blog.pyPosts, listings, tags, feed. Renders each markdown post to a pretty URL, splits current from archive writing, generates the per-tag archive pages, the RSS feed, and the sitemap. KaTeX, Mermaid, and a syntax tokenizer load from a CDN only on posts that actually use them.
build_portfolio.pyHomepage insertions. Injects the writing-cadence sparkline, the featured-and-tiles writing list, and the Publications block. Citation counts come from Semantic Scholar, cached back into the source YAML so a rate limit never wipes a number.
build_resume.pyResume and CV. One source per document, two outputs each: a print-ready PDF via WeasyPrint and a web page. Publications are rendered from the same YAML the homepage uses, so the count is consistent across all three surfaces.
build_cliff.pyThe Stars cliff curve. Renders the Medicare Advantage Star Ratings density curve on the homepage directly from the canonical CMS distribution, rather than a hand-drawn approximation. Re-run when CMS publishes the new ratings.
build_og.pySocial card. Renders the Open Graph preview image from a small composition rule, using the same locked palette tokens as the rest of the site.

Quality gates

The thing I am most deliberate about is that the site cannot quietly drift. Six linters run on every push, installed automatically the first time any project script runs, so a fresh clone wires up its own pre-push hook with no manual setup. They enforce consistency a human reviewer would eventually miss.

lint_facts.pyCross-checks employer, title, and dates across the resume, the homepage, and the structured JSON-LD metadata. The same fact stated in three places must agree.
lint_recognition.pyEvery award or service entry shown on the homepage must have a counterpart in the comprehensive CV record. The public highlight reel is a strict subset of the full one.
lint_gantt.pyThe hand-coded Education and Service timeline figure must carry a mark for every entry in the prose sections it summarizes, read back from the chart's own coordinate transform.
lint_notes.pyA margin note may not restate a number or a phrase already in the page's prose, so the asides stay additive rather than redundant.
lint_vocab.pyProgram names from CMS, like Star Ratings and Medicare Advantage, render in their canonical capitalization across every post and page.
lint_blog.pyCatches the source-side markdown mistakes that render as garbage: leaked HTML comments, blank lines inside an inline SVG, a diagram written as a blockquote.

Alongside the linters, the pre-push hook also checks that the site chrome stays free of em dashes, that the accent color stays within its strict use budget, and that the design-critique pipeline keeps no dependency on any one model vendor. A separate, scheduled pipeline runs a structured multi-camp design critique of the site against its own taste rules.

Why it is built this way

None of this is necessary for a personal website. I built it this way on purpose, because the habits are the same ones that matter in a production data system: one source of truth for every fact, generated artifacts that are reproducible from that source, automated checks that fail loudly before anything ships, and graceful degradation when an external dependency misbehaves. A portfolio that claims those habits should be able to show them. This page, and the repository behind it, are the evidence.

If you want the longer story of how the site got here, across three rebuilds and a great deal of doing things the slow way, it is written up on the blog.
